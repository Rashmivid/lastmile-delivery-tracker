## System Design Write-Up: LastMile Delivery Tracker

## Rate Calculation Engine

The rate engine (rate_engine.py) is the core business-logic component, designed around a single principle: zero hardcoded pricing. Every input to the calculation — zones, pincode-to-zone mappings, and per-combination rates — lives in the database, fully admin-configurable through dedicated CRUD endpoints. The calculation itself runs as a deterministic pipeline: detect pickup and drop zones from submitted pincodes, determine intra- vs inter-zone status, compute volumetric weight using the standard courier formula (L×B×H)/5000, bill on the higher of actual or volumetric weight, look up the matching rate card by (zone, order_type, intra/inter), apply the base charge, then conditionally add a COD surcharge. Each step is a pure function with explicit inputs and outputs, making the logic easy to test and reason about independently of the API layer. Critically, the same function is invoked twice in the order flow: once for /orders/preview (read-only, shown to the customer before confirmation) and again inside /orders/ at the moment of order creation. This dual-invocation guards against price tampering — the backend never trusts a charge value sent from the client; it always recomputes from the raw inputs (pincodes, dimensions, weight, order type, payment type) at save time.

## Zone Detection Approach

Zone detection deliberately avoids any form of geocoding or external mapping API, both to keep the system dependency-light and because the assignment's evaluation criteria call for admin-configurable, non-hardcoded logic rather than a GPS-based solution. Instead, zones are abstract admin-defined regions (e.g., "Zone A"), and individual pincodes are explicitly mapped to a zone via a dedicated pincodes table. When an order is placed, the pickup and drop pincodes are each looked up against this table to resolve their zone. If a pincode has no mapping, the system fails fast with a descriptive error rather than guessing or defaulting — this was an intentional design choice to prevent silent mispricing. The tradeoff is that this approach requires admin data entry to cover the actual service area before the system is usable end-to-end, but it keeps zone boundaries fully within admin control and trivially reconfigurable without touching code.

## Auto-Assignment Logic

Since the system has no live GPS tracking for agents, auto-assignment uses an availability-based heuristic rather than true nearest-neighbor routing. When triggered (PATCH /admin/orders/{id}/auto-assign), the system first identifies all agents currently holding an order in any "active" status (assigned, picked_up, in_transit, out_for_delivery) — these agents are considered busy. It then queries for any agent with role agent, is_active = true, and not present in that busy set, and assigns the first match. This models real-world agent capacity constraints (one active delivery at a time per agent) without requiring location infrastructure. Admins retain a manual override (/admin/orders/{id}/assign) to pick a specific agent directly when they have out-of-band knowledge (e.g., an agent's actual current location) that the system doesn't capture. This layered approach — automated default plus manual override — balances operational efficiency with human judgment where the data model has genuine limitations.
Order Status Lifecycle and Immutable Tracking

Order status follows a strict linear state machine: pending → assigned → picked_up → in_transit → out_for_delivery → delivered | failed. This is enforced at the agent level via an explicit transition map in agent.py — an agent can only move an order to the next valid state, preventing accidental or malicious status skipping (e.g., jumping straight from assigned to delivered). Admins, by contrast, have unrestricted override capability, reflecting their operational authority to correct mistakes or handle edge cases outside the normal flow. Every status change — regardless of actor — inserts a new row into a separate order_tracking table rather than mutating the order's status in place. This table is append-only by design: rows carry a timestamp, the acting user's ID and role, and an optional note, and are never updated or deleted. This produces a complete, chronological, tamper-evident audit trail per order, which is both an evaluation requirement and good practice for any system handling financial/logistics transactions where after-the-fact disputes are possible.

## Failed Delivery Handling

A failed status is a terminal state from the agent's perspective but not from the customer's. The system exposes a dedicated reschedule endpoint (PATCH /orders/{id}/reschedule) that is only accessible when an order's current status is exactly failed — enforced server-side regardless of what the frontend UI shows. On reschedule, the order's status resets to pending, its agent_id is cleared (since the original agent's attempt failed and the order re-enters the assignment pool), and the customer-provided reschedule_date is stored. This design deliberately mirrors the original order lifecycle rather than introducing new states, keeping the state machine simple. Each stage of this flow — the failure, the reschedule request, and the eventual reassignment — is logged in the tracking table, and the customer receives an email notification at each transition via the Resend integration, closing the loop on communication without requiring the customer to actively check the app.
